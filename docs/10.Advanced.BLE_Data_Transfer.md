# 10. Advanced: BLE Data Transfer & Chunking (Mule Mode Fallback)

> **Status:** Design Spec (not yet implemented)
> **Role:** Emergency fallback for media sync when Wi-Fi is unavailable.
> **Primary Media Transport:** Wi-Fi → Nodus.API → MongoDB. **This document describes the BLE fallback only.**
> **Trigger Condition:** Judge is in Mule Mode (no BLE signal from Server or any LINK node for > 10 min, Decision #23) **AND** is physically close to Admin laptop (RSSI > -60 dBm).

## 0. Transport Priority

| Priority | Transport | Condition | Used For |
|:---------|:----------|:----------|:---------|
| **1 (Primary)** | Wi-Fi → Nodus.API | Wi-Fi available | All media (photos, video, audio) |
| **2 (Fallback)** | BLE Chunked Transfer | Mule Mode + RSSI > -60 dBm | Media files only, proximity required |

> Vote scores (JSON, small payload) always travel via the BLE Swarm regardless of Wi-Fi availability.

## 1. Application-Layer Chunking (BLE Fallback)

Since we cannot rely on the OS to handle large packets automatically across all Android/iOS versions reliably, we implement an **Application-Layer Chunking** protocol locally.

### A. Packet Structure

> **Source of Truth:** The canonical byte-prefix table lives in [Doc 02 §4](02.Network.Swarm_Protocol.md). The list below is reproduced here for reference only — if there is any discrepancy, Doc 02 §4 wins.

We use a simple 1-byte prefix to distinguish packet types:

| Prefix | Type | Direction | Description |
|:-------|:-----|:----------|:------------|
| `0x01` | Vote / JSON data | Judge → Admin | Encrypted vote payload (append-only) |
| `0x02` | Media data | Judge → Admin | Photo/audio chunk (Mule Mode only) |
| `0x03` | Bootstrap data | Admin → Judge | `event_bootstrap`: full project list + rubric (Decision #27) |
| `0x04` | Project sync | Admin → Judge | `project_sync`: delta of new projects since last pull (Decision #29, #42) |
| `0x05` | Blocklist sync | Admin → LINK nodes | `blocklist_sync`: updated blocked-key list broadcast via `NODUS_ACK_NOTIFY` (Decision #41) |
| `0xA1` | ACK | Admin → Judge | Packet acknowledgement |

### B. The Sequence (Media Sync)

1.  **Client** discovers Server and connects.
2.  **Client** monitors RSSI. When in Mule Mode AND RSSI > -60 dBm (Admin proximity confirmed), media sync triggers.
3.  **Client** compresses image (Goal: ~512KB).
4.  **Client** constructs payload: `[0x02][VoteId(16)][ImageBytes...]`.
5.  **Client** splits payload into chunks using `ChunkerService`.
6.  **Client** sends chunks sequentially with 20ms throttling.
7.  **Server** reassembles chunks using `ChunkAssembler`.
8.  **Server** saves image to disk.
9.  **Server** sends ACK Notification: `[0xA1][VoteId(16)]`.
10. **Client** receives ACK and marks `MediaStatus = ReachedAdmin`. The Admin node stores the file locally; when internet becomes available the Admin syncs to MongoDB GridFS and `MediaStatus` advances to `ReachedCloud`.

## 2. MTU Strategy

- **Default MTU:** 180 bytes (Safe minimum for iOS/Android compatibility).
- **Android:** Can request MTU = 512, but we stick to 180 for consistency.

## 3. Implementation Details

### Client Side (`MediaSyncService.cs`)

- Uses `IBleClientService.ReadRssiAsync` to poll signal strength.
- Uses `ChunkerService.Split` to generate chunks.
- Subscribes to `IBleClientService.Notifications` to receive ACKs.

### Server Side (`BleServerService.cs`)

- Uses `ChunkAssembler` to handle incoming streams.
- Writes to `AppData/Media/{EventId}/{VoteId}.jpg`.
- Sends ACK notification upon successful save.
