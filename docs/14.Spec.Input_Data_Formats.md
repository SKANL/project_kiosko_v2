# 14. Spec: Registration & Rubric Standards

> **Status:** Active
> **Purpose:** Detailed requirements for Data Input (Self-Service) and Config.

## 1. Student Registration Input (The Form)

Students register themselves. The System (Server) creates the record.

### Required Fields (Student)

| Field              | Type     | Min/Max Rules     | Description                 |
| :----------------- | :------- | :---------------- | :-------------------------- |
| **`Project Name`** | Text     | 3-64 chars        | The official title.         |
| **`Category`**     | Dropdown | One of Config key | E.g., "Software", "Social". |

### Optional Fields (Student)

| Field             | Type | Rules         | Description            |
| :---------------- | :--- | :------------ | :--------------------- |
| **`Members`**     | Text | Max 128 chars | Names of team members. |
| **`Github Link`** | URL  | Valid HTTPS   | Link to source code.   |
| **`Description`** | Text | Max 140 chars | Twitter-style pitch.   |

### System Generated (ReadOnly)

- **`Project ID`:** **Canonical format: `PROJ-{3_ALPHANUMERIC_UPPERCASE}`** (e.g., `PROJ-A3X`, `PROJ-B7K`). Generated as the next available sequential-random code. This format is used everywhere in the system — no other format is valid.
- **`QR Payload`:** `nodus://vote?pid=PROJ-A3X`
- **`Edit Token`:** A single-use UUID generated at registration time. Allows the student to reopen their registration form via `http://[LOCAL_IP]:5000/edit?token={Edit_Token}` to correct typos before the event starts. Token does not expire until the event is closed by the Admin.

### API: Event Configuration (Consumed by Nodus.Web)

Before rendering the registration form, Nodus.Web calls this endpoint to populate the Category dropdown and display the event name (Decision #19):

```
GET http://[LOCAL_IP]:5000/api/event/current
```

**Response:**
```json
{
  "eventId": "EVT-XYZ",
  "name": "Hackathon 2026",
  "categories": ["Software", "Hardware", "Social"],
  "maxProjects": 100
}
```

This endpoint is served by Nodus.Admin's embedded HTTP server. No internet required — it runs exclusively on the local Wi-Fi network.

> **`maxProjects` enforcement (Decision #49):** When the number of registered projects reaches `maxProjects`, Nodus.Web displays a **"Registration closed — maximum number of projects reached"** banner and disables the registration form. The Admin endpoint returns `HTTP 409 Conflict` for any subsequent `POST /api/projects` call. The Admin can raise the limit at any time from the Dashboard settings.

## 2. Input: Rubric Configuration (The Scoring Rules)

Admin defines "what to judge" via JSON or UI.

> **⚠️ Rubric Lock (Decision #46):** The rubric is **immutable** once the first Vote exists in the database for this event. The Admin UI prevents edits at that point with the message: _"Rubric cannot be changed after voting has started."_ Configure the rubric completely before opening the event.

### Structure (Stored in `Event.RubricJson`)

```json
{
  "criteria": [
    {
      "id": "c1",
      "label": "Originality",
      "description": "How novel is the idea?",
      "min": 1,
      "max": 10,
      "weight": 1.0,
      "step": 1
    },
    {
      "id": "c2",
      "label": "Execution",
      "description": "Does it crash?",
      "min": 0,
      "max": 5,
      "weight": 2.0,
      "step": 0.5
    }
  ]
}
```

## 3. Output: Results Export (`Final_Results.xlsx`)

Generated for the Award Ceremony.

### Scoring Formula (Decision #55)

Before detailing the sheet layout, here is the canonical formula used to compute all scores:

**Step 1 — Score por juez (por proyecto):**

Given a vote with per-criterion scores `{c1: s1, c2: s2, ...}` and rubric weights `{c1: w1, c2: w2, ...}`:

$$\text{judgeScore} = \frac{\sum_i (s_i \times w_i)}{\sum_i w_i}$$

This produces a weighted score normalized to the rubric’s value range — independent of how many criteria exist or how they are weighted.

**Step 2 — Average Score del proyecto:**

$$\text{AverageScore} = \frac{1}{N_{\text{judges}}} \sum_{j=1}^{N_{\text{judges}}} \text{judgeScore}_j$$

Only the vote with the **highest `version`** per `(judge_id, project_id)` pair is used (Latest Version Wins — Decision #7).

**Step 3 — Total Points del proyecto (tie-breaker):**

$$\text{TotalPoints} = \sum_{j=1}^{N_{\text{judges}}} \sum_i (s_{ji} \times w_i)$$

Raw weighted sum across all judges, NOT normalized. Projects with the same rounded `AverageScore` are ranked by `TotalPoints` descending.

> **Ejemplo con la rúbrica de §2:** c1 (Originality, weight=1.0, max=10), c2 (Execution, weight=2.0, max=5).
> A judge votes `{c1: 8, c2: 4}`: `judgeScore = (8×1.0 + 4×2.0) / (1.0+2.0) = 16/3 = 5.33`.
> This is normalized to the collective rubric range, not to any single criterion’s max.
