# 15. UX: Student Registration (Self-Service)

> **Status:** Active
> **Role:** Student / Project Leader
> **Goal:** Frictionless Onboarding (No Admin Data Entry)

## 1. The Entry Point

1.  **Context:** The Organizer projects a giant **"REGISTRATION QR"** on the auditorium screen.
2.  **Requirement:** The venue must have a **local Wi-Fi network** (router provided by the organizer — see Runbook Doc 07). Internet is not required.
3.  **Action:** Student scans the QR with their standard Camera App.
4.  **Destination:** Opens the **Nodus.Web** PWA in the browser: `http://[LOCAL_IP]:5000/register?event=EVT-ID`

> **Architecture Note:** Nodus.Web is **100% online** (requires the local Wi-Fi network). It is a Blazor WebAssembly app served by the Nodus.Admin local HTTP server. There is no offline/LocalStorage fallback for student registration — if the student cannot connect to the local network, they cannot register.

## 2. The Interaction (The Form)

We ask ONLY for what matters. The System handles the bureaucracy.

### Fields

| Field            | Type     | Required | UX Hint                                            |
| :--------------- | :------- | :------- | :------------------------------------------------- |
| **Project Name** | Text     | ✅       | "EcoSpark Energy"                                  |
| **Category**     | Dropdown | ✅       | [Software, Hardware, Social] (Configured by Admin) |
| **Description**  | TextArea | ❌       | "Max 140 chars pitch"                              |
| **GitHub/Link**  | URL      | ❌       | "Where's the code?"                                |
| **Team Members** | Tags     | ❌       | "Juan, Maria..."                                   |

### Automation (What the Student doesn't see)

- **Auto-ID:** System assigns the next available `PROJ-{3_ALPHANUMERIC_UPPERCASE}` code (e.g., `PROJ-A3X`). **This is the canonical format — no other format is used anywhere in the system.**
- **Stand Number:** System assigns `A-12` (next available slot) OR `null` (displays as "Unassigned") — the Admin can reassign from the dashboard at any time (Decision #26).
- **Edit Token:** System generates a unique UUID `Edit_Token`. The student's "Boarding Pass" screen shows a link `http://[LOCAL_IP]:5000/edit?token={Edit_Token}` so they can correct typos before the event starts. The token remains valid until the Admin closes the event. See [Doc 14](14.Spec.Input_Data_Formats.md) for the full field spec.

## 3. The "Ticket" (Visual Confirmation)

Once submitted, the Student gets a digital "Boarding Pass" on their phone.

### The Screen Shows:

- **Big QR Code:** Contains `nodus://vote?pid=PROJ-A3X`. This is what Judges will scan!
- **Project ID:** **PROJ-A3X** (Huge Text).
- **Stand:** "Go to Table A-12".
- **Instruction:** "Place your phone or print this QR on your table."

---

## 4. The Loop

1.  Student registers.
2.  Student gets QR.
3.  Judge scans Student's QR.
4.  Voting begins.

_Zero Admin Excel Work._
