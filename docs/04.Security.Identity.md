# 04. Security & Identity

> **Status:** Draft
> **Key Concept:** Trustless Offline Verification

## 1. Identity Management

In an offline environment, we cannot hit an Auth0 or Firebase API to verify a token. We use a **Bootstrap Offline PKI** model: the event password is the root of trust, and judge identity is established on first contact.

> **Decision:** No pre-registration is required. Judges do not need internet access before the event. The `SharedAESKey` embedded in the Judge Access QR is the primary authentication mechanism.

### A. The Handshake (Offline Event Day — "Trust On First Use")

When the Judge arrives at the event:

1.  **Launch:** Judge opens Nodus.Judge app.
2.  **Scan Judge Access QR:** Displayed on the projector by the Admin.
    - **QR Content:** `AES_Encrypt({SharedAESKey, ServerUUID, EventID}, PBKDF2(EventPassword, EventId))` (Decision #22)
3.  **Enter Event Password:** Provided orally by the organizer.
    - App decrypts payload → Obtains `SharedAESKey`, `ServerUUID`, `EventID`.
    - Without the password, the QR is cryptographically useless.
4.  **Key Generation (On-Device):** App generates an `Ed25519` key pair locally.
    - Private Key → stored in device Secure Storage (Keystore / Keychain).
    - Public Key → included in the **Registration Packet** (first BLE packet sent).
5.  **Registration Packet:** The app sends a `type: "judge_register"` packet **via direct GATT connection to the Admin node** — not through the Swarm. The Swarm may not be established yet when the first judge arrives (Decision #17):
    ```json
    {
      "type": "judge_register",
      "judge_id": "UUID-V4-GENERATED-ON-DEVICE",
      "judge_name": "Dr. Brown",
      "public_key": "BASE64_ED25519_PUBLIC_KEY",
      "event_id": "EVT-XYZ"
    }
    ```
    The `judge_id` is a UUID V4 generated on the Judge device **before** sending this packet (Decision #35). It becomes the `JudgeId` on every `Vote` record this judge creates — no server-issued ID is needed. The packet is encrypted with `SharedAESKey` (proving the judge has the event password).
6.  **Server Registers the Judge + Bootstrap:** The Admin node:
    - Stores the judge’s `judge_id`, name, and public key in the `Judges` table. All subsequent vote packets from this judge are verified against this key.
    - Exposes the `event_bootstrap` payload (full project list + `RubricJson`) on the `NODUS_BOOTSTRAP_READ` characteristic. **The payload is AES-encrypted with `SharedAESKey` before being written to the characteristic buffer, so an observer receiving BLE notifications cannot read the project list or rubric** (Decision #44). The Judge reads it in 180-byte pages using `ATT_READ_BLOB_REQ` paging until the payload is fully assembled (Decision #36, #27). The Judge stores the result in local SQLite — enabling fully offline voting immediately (Decision #18).
    - After this exchange, the judge disconnects from the direct GATT connection and enters the Swarm normally for vote transmission.
    - **Project delta (post-onboarding):** The Judge app polls `NODUS_BOOTSTRAP_READ` every **`Random(50s, 70s)`** (jittered to prevent thundering-herd — Decision #43) during the active event. The Admin responds with a `project_sync` packet (prefix `0x04`) containing only projects registered after the judge's last pull. The Judge merges them into local SQLite automatically. No persistent Admin–Judge connection is required (Decision #29).

### B. Admin: Judge Blocking / Expulsion

If the Admin identifies an unknown or malicious judge (e.g., someone who obtained the password but was not invited):

1.  **Admin UI:** Admin opens the "Node Inspector" panel and identifies the suspicious judge.
2.  **Block Action:** Admin clicks "Block Judge" → Server adds the judge's `public_key` to a **Blocklist**.
3.  **Effect:**
    - All packets signed with that key are **silently rejected** by the Server.
    - The Server emits a `blocklist_sync` packet (prefix `0x05`) as a GATT Notification on `NODUS_ACK_NOTIFY`. Any LINK node currently in active relay state will receive it and begin dropping packets from blocked keys during their remaining LINK window (Decision #41 — see [Doc 02 §4](02.Network.Swarm_Protocol.md) for the canonical packet format).
    - **Limitation (best-effort):** LINK nodes are ephemeral (max 60s). A blocklist broadcast only reaches nodes that are in LINK state at the moment of transmission. Nodes in SEEKER or COOLDOWN state will not have an updated blocklist until they re-enter LINK. The Server is the **authoritative** and final enforcement point — a blocked packet that slips through a relay is still rejected at the Server.
4.  **No re-admission** without a new event password and fresh key generation.

## 2. QR Code Strategy (The Physical Link)

We use QR codes not just for URLs, but as **Data Carriers**.

### Canonical QR Code Taxonomy

The system uses exactly **3 types of QR codes**. All other names in older docs are superseded by this table:

| Name | Generated by | Scanned by | Content |
|:-----|:------------|:-----------|:--------|
| **Event Registration QR** | Admin (at event start) | Students (phone camera) | `http://[LOCAL_IP]:5000/register?event=EVT-ID` |
| **Judge Access QR** | Admin (once per event) | Judges (Nodus.Judge app) | `AES_Encrypt({SharedAESKey, ServerUUID, EventID}, PBKDF2(Password, EventId))` (Decision #22) |
| **Project Display QR** | Auto-generated per project | Judges (Nodus.Judge app, to vote) | `nodus://vote?pid=PROJ-A3X` |

**Canonical Project ID Format:** `PROJ-{3_ALPHANUMERIC_UPPERCASE}` (e.g., `PROJ-A3X`, `PROJ-B7K`).

- **NO Internet required** to read a Project Display QR: the Judge app uses it as a fast lookup key against its local project cache.

## 3. Anti-Cheat & Integrity

How do we prevent a student from capturing a Bluetooth packet of a "100/100" score and replaying it 50 times?

### A. Envelope Signing

Every vote payload is signed with the Judge's **Private Key**.
`Signature = Ed25519_Sign(Payload + Timestamp, PrivateKey)`

The Server (which has the Public Key) verifies: "Yes, this really came from Judge Alice, and nobody tampered with the score."

### B. Anti-Replay (Nonce & caching)

1.  **Nonce:** Every packet includes a unique GUID and a Unix Timestamp.
2.  **Server Logic:** The Server maintains a "Seen Packet Cache".
    - If `PacketID` exists in cache -> **REJECT** (Duplicate).
    - If `Timestamp` is older than 2 hours -> **REJECT** (Old/Stale).

### C. Loop Prevention (The Relay Defense)

**Critical for Firefly Protocol:**
Every Relay Node (Link) MUST utilize a short-term **Bloom Filter** or `MemoryCache` of `packet_id`s seen in the last **10 minutes** (canonical per [Doc 02](02.Network.Swarm_Protocol.md) and Decision #15).

- **Rule:** If I have already relayed Packet X, I MUST NOT relay it again.
- **Why?** Prevents "Echo Chambers" where two nodes bounce the same packet back and forth forever.

### D. Encryption (Privacy)

We don't want Students sniffing scores.
Result: `AES_Encrypt(JSON + Signature, Shared_Event_Key)`

- Only the Server (and other Judges with the Event QR) can decrypt the traffic.
- To an outsider, it looks like random noise.

## 4. Canonical Vote Payload Format (Pre-Encryption) (Decision #51)

Before AES encryption, the vote payload that a Judge constructs and signs is:

```json
{
  "packet_id": "UUID-V4",
  "type": "vote",
  "judge_id": "UUID-V4-OF-JUDGE",
  "event_id": "EVT-XYZ",
  "project_id": "PROJ-A3X",
  "version": 1,
  "scores": {
    "c1": 8,
    "c2": 4.5
  },
  "comment": "Great execution, needs polish on the UI.",
  "timestamp_utc": 1741046400,
  "local_photo_path": null,
  "local_audio_path": null,
  "signature": "BASE64_ED25519_SIGNATURE"
}
```

### Field Definitions

| Field | Type | Description |
|:------|:-----|:------------|
| `packet_id` | UUID V4 | Unique per packet; used by Admin Bloom Filter for dedup (Decision #15). |
| `type` | string | Always `"vote"` for vote packets. |
| `judge_id` | UUID V4 | The judge's permanent ID generated at registration (Decision #35). |
| `event_id` | string | FK to the event; used for scoping on the server. |
| `project_id` | string | `PROJ-{3_ALPHANUM}` format (Decision #6). |
| `version` | int | `1` on first submission; incremented on re-evaluation. Server applies "Latest Version Wins" (Decision #7, #34). |
| `scores` | object | Keys are rubric `criterion.id` values; values are numbers within `[min, max]` (Decision #46 rubric). |
| `comment` | string? | Optional free-text annotation by the judge. Max 500 chars. Null if omitted. |
| `timestamp_utc` | int | Unix epoch (seconds) at submission time. Used for anti-replay: packets older than 2 hours are rejected (Doc 04 §3B). |
| `local_photo_path` | string? | Device-local path of attached photo, or `null` (Decision #20, #47). |
| `local_audio_path` | string? | Device-local path of attached audio, or `null` (Decision #20, #47). |
| `signature` | Base64 string | `Ed25519_Sign(canonical_json_without_signature_field, PrivateKey)`. See signing rule below. |

### Signing Rule

1.  Construct the JSON object **without** the `signature` field.
2.  Serialize to a deterministic UTF-8 string (keys sorted alphabetically).
3.  Sign the UTF-8 bytes with the judge's `Ed25519` private key.
4.  Set `signature` to the Base64-encoded result.
5.  AES-encrypt the entire JSON string (including `signature`) with `SharedAESKey`.
6.  Wrap in the transport envelope:
    ```json
    { "packet_id": "...", "hops": [], "payload": "AES_BASE64_CIPHER", "ttl": 2 }
    ```

> **Why sort keys?** Different JSON serializers emit fields in different orders. Sorting guarantees the Judge and Admin produce byte-identical strings for the same logical vote, making signature verification deterministic.
