# 00. Decisions Log — Nodus Architecture

> **Propósito:** Registro permanente de todas las decisiones arquitectónicas tomadas, con fecha y razonamiento. Es la fuente canónica cuando un documento existente contradiga algo aquí.
> **Última actualización:** 2026-03-04 (Decisiones #54–#56 añadidas; #3/#25/#39/#40 corregidas)

## DECISIÓN #1 — Plataforma del Admin: Windows Desktop

**Fecha:** 2026-03-03
**Decisión:** Nodus.Admin es una aplicación **Windows Desktop** (MAUI con target Windows / WinUI 3).
**Razonamiento:** El nodo Admin es el GATT Server permanente (nunca entra en cooldown). Un laptop Windows ofrece mayor estabilidad de Bluetooth, no tiene restricciones de iOS/Android background, y puede servir simultáneamente como servidor HTTP local para el registro de estudiantes. El organizador opera desde una "mesa de comando".
**Documentos afectados:** 01, 05, 07, 13.

---

## DECISIÓN #2 — Sin Librería Compartida (No Nodus.Shared)

**Fecha:** 2026-03-03
**Decisión:** No existe `Nodus.Shared`. Cada app (Nodus.Admin, Nodus.Judge, Nodus.Web, Nodus.API) es completamente independiente y posee sus propios modelos y servicios.
**Razonamiento:** Las apps tienen ciclos de vida distintos, plataformas distintas y necesidades de despliegue independientes. Una librería compartida crea acoplamiento frágil entre proyectos que deben poder evolucionar separadamente.
**Documentos afectados:** 01, 05.

---

## DECISIÓN #3 — FSM Canónico: Doc 02 es la única fuente de verdad

**Fecha:** 2026-03-03
**Decisión:** El protocolo Firefly State Machine está definido exclusivamente en [Doc 02](02.Network.Swarm_Protocol.md). Cualquier definición en otros documentos es un artefacto histórico y debe ignorarse.

**Parámetros canónicos:**

| Parámetro | Valor |
|:----------|:------|
| RSSI → CANDIDATE threshold | `-75 dBm` |
| Random wait (Trickle) | `Random(3s, 15s)` |
| Max LINK duration | `60s` |
| COOLDOWN duration | `5 min` |
| Battery veto | `< 20%` (en código: `< 0.20` — ver Decisión #53) |
| Redundancy constant (k) | `2` links activos cercanos |

**Nombres de estados canónicos:** `SEEKER`, `CANDIDATE`, `LINK`, `COOLDOWN`.
**Nomenclatura obsoleta (no usar):** IDLE, LOST, HERO, TIRED.
**Documentos afectados:** 02, 09, 12.

## DECISIÓN #4 — Semántica de TTL: Cuenta Relays Intermedios

**Fecha:** 2026-03-03
**Decisión:** `ttl` en el JSON de paquete = **número de nodos relay intermedios permitidos** (el Judge origen y el Server destino NO se cuentan).
**Resultado:** `ttl=2` → cadena máxima: `Judge → Relay1 → Relay2 → Server` (2 relays).
**Razonamiento:** Consistente con el diagrama de Doc 12 §3C y suficiente cobertura para la mayoría de escenarios de sala.
**Documentos afectados:** 02.

---

## DECISIÓN #5 — Transporte de Media: Wi-Fi Primario, BLE Fallback

**Fecha:** 2026-03-03
**Decisión:**
- **Primario:** Fotos, videos y audio se transfieren vía Wi-Fi → Nodus.API → MongoDB GridFS.
- **Fallback (Mule Mode):** Si no hay Wi-Fi disponible Y el Judge está físicamente cerca del Admin laptop (RSSI > -60 dBm), se activa la transferencia BLE chunked (ver Doc 10).
- **Datos de votación (JSON scores):** Siempre viajan por BLE Swarm, sin importar el estado de Wi-Fi.

**Razonamiento:** Wi-Fi es más eficiente para archivos grandes. BLE en Mule Mode garantiza que ningún dato se pierda incluso sin internet, pero requiere proximidad física (lo cual es aceptable para un escenario de feria).
**Documentos afectados:** 03, 10.

## DECISIÓN #6 — Formato Canónico de Project ID

**Fecha:** 2026-03-03
**Decisión:** El único formato válido de Project ID es: **`PROJ-{3_ALFANUM_MAYUSC}`**

**Ejemplos válidos:** `PROJ-A3X`, `PROJ-B7K`, `PROJ-Z9Q`
**Formatos obsoletos (NO usar):** `PROJECT_004`, `P-001`, `PROJ-023` (3 dígitos), etc.

**QR payload canónico:** `nodus://vote?pid=PROJ-A3X`

**Razonamiento:** Un único formato elimina bugs de parsing. 3 caracteres alfanuméricos en mayúscula = 36³ = 46,656 IDs posibles, más que suficiente para cualquier evento.
**Documentos afectados:** 03, 04, 14, 15.

## DECISIÓN #7 — Estrategia de Re-evaluación de Votos

**Fecha:** 2026-03-03
**Decisión:**
- La base de datos es **append-only**. Los votos nunca se modifican en-place.
- Cuando un juez re-evalúa un proyecto, se crea un **nuevo registro** con `version` incrementado.
- El servidor aplica **"Latest Version Wins"** basado en el campo `version` (el registro con mayor `version` gana). `CreatedAtUtc` se conserva únicamente para auditoría (Decision #34).
- La pantalla del juez se llama **"Re-evaluate"** (o **"Update Evaluation"** en la UI). Muestra los scores previos pre-cargados.

**Razonamiento:** Append-only simplifica la sincronización distribuida — no hay conflictos de merge. El historial completo de evaluaciones queda preservado para auditoría.
**Documentos afectados:** 03, 06.

## DECISIÓN #8 — Seguridad: Bootstrap Offline PKI con Bloqueo de Jueces

**Fecha:** 2026-03-03
**Decisión:** No se requiere pre-registro online. El flujo es:

1. Admin genera el **Judge Access QR** con la contraseña del evento.
2. Juez escanea QR + ingresa contraseña → obtiene `SharedAESKey`.
3. App genera par de llaves `Ed25519` **en el dispositivo** sin internet.
4. Primer paquete BLE = `judge_register` con clave pública, cifrado con `SharedAESKey`.
5. Server registra la clave (Trust On First Use).

**Bloqueo de jueces:** Admin puede bloquear cualquier juez desde el Node Inspector. La clave pública del juez bloqueado se añade al Blocklist. Todos los paquetes firmados con esa clave son rechazados (por el servidor y por los nodos relay).

**Razonamiento:** Máxima flexibilidad operacional (jueces de último minuto pueden unirse sin internet previo). La contraseña del evento + cifrado AES es la barrera de entrada. El Blocklist añade control posterior si se detecta actividad sospechosa.
**Documentos afectados:** 04, 06.

## DECISIÓN #9 — Nodus.Web: HTTP/Wi-Fi Only, Sin BLE

**Fecha:** 2026-03-03
**Decisión:** Nodus.Web (app de estudiantes) usa **exclusivamente HTTP sobre Wi-Fi**. No participa en el swarm BLE.
**Razonamiento:** Web Bluetooth no está soportado en Safari/iOS ni Firefox, cubriendo ~80% de los dispositivos de los estudiantes. La web es la tecnología más accesible (no requiere instalación). Los estudiantes solo necesitan registrar su proyecto y mostrar un QR — no necesitan BLE para esto.
**Documentos afectados:** 01, 09, 15.

---

## DECISIÓN #10 — Red Local Requerida para Registro de Estudiantes

**Fecha:** 2026-03-03
**Decisión:** Nodus.Web es **100% online** (requiere conexión a la red Wi-Fi local del evento). No hay modo offline para estudiantes.
- URL del QR de registro: `http://[LOCAL_IP]:5000/register?event=EVT-ID` (NO un dominio público).
- Nodus.Admin sirve Nodus.Web vía un servidor HTTP local embebido.
- El router Wi-Fi local es un **requisito obligatorio** del evento (no opcional).

**Razonamiento:** Los estudiantes solo interactúan con el sistema una vez (al inicio). Si no tienen Wi-Fi local en ese momento, el organizador puede registrarlos manualmente desde el Admin. La complejidad de hacer Nodus.Web offline-capable no justifica el beneficio.
**Documentos afectados:** 07, 15.

## DECISIÓN #11 — Sin Bridges Fijos en Runbook

**Fecha:** 2026-03-03
**Decisión:** No existen teléfonos "dedicados como bridges" en el hardware del evento. Si se llevan dispositivos extra, son **"Spare Judge Devices"** que participan en el swarm automáticamente.
**Razonamiento:** El protocolo Firefly es dinámico — ningún nodo tiene un rol permanente. Intentar "configurar" un bridge fijo no es posible en la implementación actual y causaría confusión operacional.
**Documentos afectados:** 07.

**Fecha:** 2026-03-03

## DECISIÓN #14 — Online-First para Nodus.Web

**Fecha:** 2026-03-03
**Decisión:** Nodus.Web no necesita almacenamiento local (no LocalStorage, no IndexedDB). Es completamente stateless en el cliente — el estado vive en el servidor (Nodus.Admin + Nodus.API).
**Razonamiento:** Si el estudiante cierra el browser antes de registrarse, simplemente vuelve a abrir la URL y empieza de nuevo. No hay datos críticos que preservar en el cliente.
**Documentos afectados:** 03 (nota), 15, 17 (implied).

## DECISIÓN #15 — Bloom Filter TTL: 10 Minutos

**Fecha:** 2026-03-04
**Decisión:** El Bloom Filter en nodos relay retiene `packet_id`s por **10 minutos**.
**Razonamiento:** Doc 02 es la única fuente de verdad (Decisión #3). Con `ttl=2` y cadenas de hasta 3 saltos en entornos congestionados, un paquete puede tardar varios minutos en llegar. Un TTL de 60 segundos (valor incorrecto que aparecía en Doc 04 §3C) es insuficiente y causaría re-relay de paquetes aún en tránsito, generando duplicados.
**Documentos afectados:** 04.

---

## DECISIÓN #16 — Heartbeat FSM: Cada 1-5 Segundos

**Fecha:** 2026-03-04
**Decisión:** El loop del FSM Firefly corre cada **1-5 segundos** (intervalo variable aleatorio para evitar sincronización entre nodos).
**Razonamiento:** Doc 02 es la única fuente de verdad (Decisión #3). El valor de 10 segundos que aparecía en Doc 12 §2 es demasiado lento — en un salón donde la gente se mueve, 10s de latencia para reaccionar a un cambio de señal degrada la capacidad de auto-reparación del swarm.
**Documentos afectados:** 12.

## DECISIÓN #17 — Registro del Juez: Conexión GATT Directa al Admin (No por Swarm)

**Fecha:** 2026-03-04
**Decisión:** El paquete `judge_register` se envía vía **conexión GATT directa al Admin node**, NO a través del Swarm relay.
**Razonamiento:** Cuando el primer juez llega al evento, el Swarm aún no está establecido (ningún nodo ha alcanzado estado LINK todavía). Intentar enviar `judge_register` por el Swarm causaría un deadlock: no hay relays porque no hay jueces registrados, y no hay jueces registrados porque no hay relays. El Admin siempre está publicitando `NODUS_MAIN_SERVICE` — los jueces se conectan directamente para el handshake inicial. Después de registrarse, el juez entra al Swarm normal para los votos.
**Documentos afectados:** 04, 06.

---

## DECISIÓN #18 — Catálogo de Proyectos: Bootstrap en Registro + Sync Delta por Pull

**Fecha:** 2026-03-04
**Decisión:** La respuesta del Admin al `judge_register` incluye un payload `event_bootstrap` con:
- La lista completa de proyectos registrados hasta ese momento.
- La rúbrica de evaluación (`RubricJson`).

Para proyectos registrados **después** del onboarding del juez, el Judge sondea `NODUS_BOOTSTRAP_READ` cada **`Random(50s, 70s)`** (jitter anticolisión — Decisión #43) y recibe un delta `project_sync` con solo los proyectos nuevos. **⚠️ El mecanismo original de esta decisión ("Admin emite project_sync por BLE Swarm") fue corregido por [Decisión #37](#decisión-37): `project_sync` es pull-based directo, no propagado por el Swarm.**
**Razonamiento:** Resuelve el problema de "¿cómo sabe el juez qué proyectos existen?" sin requerir Wi-Fi en el dispositivo del juez. Un juez que llega tarde recibirá proyectos nuevos vía pull periódico; mientras tanto, puede votar los proyectos que ya tiene en caché.
**Documentos afectados:** 04, 06. **Véase también:** Decisión #29, #37, #39.

## DECISIÓN #20 — Media en Votos: Completamente Opcional

**Fecha:** 2026-03-04
**Decisión:** Fotos y notas de voz en el formulario de votación son **completamente opcionales**. El formulario tiene botones "📷 Add Photo" y "🎤 Voice Note" que el juez puede ignorar. Un voto sin media es válido y completo.
**Razonamiento:** Hacer la media obligatoria bloquearía el flujo de votación en eventos con muchos proyectos (100+). El valor principal está en los scores numéricos y los comentarios textuales; la media es evidencia complementaria. Forzarla aumentaría el tiempo por voto y la frustración del juez.
**Documentos afectados:** 06.

## DECISIÓN #21 — Conteo de Jueces en Dashboard: Denominador Dinámico

## DECISIÓN #22 — KDF para Contraseña del Evento: PBKDF2 (No SHA256)

**Fecha:** 2026-03-04
**Decisión:** La clave AES para cifrar el Judge Access QR se deriva de la contraseña del evento usando **PBKDF2**, no SHA256 directamente.

**Fórmula canónica:**
```
AES_Key = PBKDF2(password: EventPassword, salt: UTF8(EventId), iterations: 100_000, hash: SHA256, keyLength: 32)
```

**Implementación .NET 10:**
```csharp
byte[] key = Rfc2898DeriveBytes.Pbkdf2(
    password:      Encoding.UTF8.GetBytes(eventPassword),
    salt:          Encoding.UTF8.GetBytes(eventId),
    iterations:    100_000,
    hashAlgorithm: HashAlgorithmName.SHA256,
    outputLength:  32);
```

**Representación en documentación** (forma corta): `PBKDF2(Password, EventId)`

**Razonamiento:** SHA256 es una función hash de propósito general (~1 ns/hash). Un atacante con el QR capturado puede probar millones de contraseñas por segundo offline. PBKDF2 con 100 000 iteraciones eleva ese costo a ~100 ms por intento, haciendo el ataque de fuerza bruta imprácticamente lento. El cálculo ocurre una sola vez al escanear el QR, por lo que no penaliza la UX del juez.
**Documentos afectados:** 00 (tabla QR en Dec #12), 04, 06, 13.

## DECISIÓN #23 — Definición Canónica de Mule Mode

**Fecha:** 2026-03-04
**Decisión:** El Judge entra en **Mule Mode** cuando **no ha detectado el Admin Server ni ningún nodo LINK** (advertising `NODUS_MAIN_SERVICE`) durante **más de 10 minutos consecutivos**.

- Mule Mode **no es** una condición de Wi-Fi o internet — es aislamiento del **BLE Swarm**.
- **Condición de salida:** El dispositivo detecta cualquier anuncio `NODUS_MAIN_SERVICE` con RSSI > -80 dBm.
- **Sub-condición (Mule Transfer activo):** Si adicionalmente el Admin Server es detectable con RSSI > -60 dBm (proximidad física), la transferencia de media BLE chunked se activa automáticamente (ver Doc 10 §1).

**Razonamiento:** Unifica las tres descripciones contradictorias de Mule Mode en Docs 03, 06 y 10. La condición correcta es aislamiento BLE, no ausencia de Wi-Fi — un juez puede tener Wi-Fi pero estar fuera del rango del swarm, o al revés.
**Documentos afectados:** 03, 06, 10.

## DECISIÓN #24 — SyncStatus: Dos Campos Independientes en Vote

**Fecha:** 2026-03-04
**Decisión:** El modelo `Vote` usa **dos campos de estado independientes** en lugar de uno:

```csharp
enum SyncStatus { Pending, ReachedAdmin, ReachedCloud, SyncError }

SyncStatus ScoreStatus {get;set;}  // Para PayloadJson (scores) — viaja por BLE Swarm
SyncStatus MediaStatus {get;set;}  // Para fotos/audio — viaja por Wi-Fi o BLE Mule Mode
```

**Mapeo de estados:**
- `ScoreStatus = ReachedAdmin` → scores llegaron al Admin via BLE Swarm.
- `ScoreStatus = ReachedCloud` → Admin sincronizó scores a MongoDB via Nodus.API.
- `MediaStatus = ReachedAdmin` → archivos llegaron al Admin (equivalente al anterior `IsMediaTransferred = true`).
- `MediaStatus = ReachedCloud` → archivos están en MongoDB GridFS (equivalente al anterior `RemotePhotoUrl != null`).

El campo `IsMediaTransferred` se elimina. El indicador de UI (TopAppBar) se basa en `ScoreStatus`: verde = `ReachedAdmin`, ámbar = `Pending`, rojo = `SyncError`.
**Documentos afectados:** 03.

## DECISIÓN #25 — GATT UUIDs Canónicos

**Fecha:** 2026-03-04
**Decisión:** Los UUIDs de servicio y características GATT son fijos y **compartidos** por el Admin GATT Server y los nodos en estado LINK (relay GATT Server). Ambos roles exponen exactamente el mismo servicio y características.

| Constante              | UUID                                   | Tipo GATT            |
|:-----------------------|:---------------------------------------|:---------------------|
| `NODUS_MAIN_SERVICE`   | `6E6F6400-0000-0000-0000-000000000001` | Primary Service      |
| `NODUS_DATA_WRITE`     | `6E6F6400-0000-0000-0000-000000000002` | WriteWithoutResponse |
| `NODUS_ACK_NOTIFY`     | `6E6F6400-0000-0000-0000-000000000003` | Notify               |
| `NODUS_BOOTSTRAP_READ` | `6E6F6400-0000-0000-0000-000000000004` | Read                 |

> Nota: `6E6F64` es la representación ASCII hex de `"nod"` (de Nodus).

**Razonamiento:** Sin UUIDs concretos, Admin y Judge uses UUID strings distintos en su código y las conexiones GATT fallan silenciosamente. Esta es la única fuente de verdad para estos valores.
**Documentos afectados:** 02, 09.

> ⚠️ **Parcialmente supersedida por Decisión #50:** La afirmación original de que «ambos roles exponen exactamente el mismo servicio y características» es **incorrecta**. La Decisión #50 define que `NODUS_BOOTSTRAP_READ` es **Admin-only**. Los UUIDs de servicio y las demás características (`NODUS_DATA_WRITE`, `NODUS_ACK_NOTIFY`) siguen siendo como se define aquí.

## DECISIÓN #27 — event_bootstrap: Usa el Protocolo de Chunking MTU

**Fecha:** 2026-03-04
**Decisión:** La respuesta `event_bootstrap` del Admin al juez (lista completa de proyectos + `RubricJson`) se transmite usando el **mismo protocolo de chunking MTU** que los paquetes de media (ver Doc 10 §1), dado que puede superar fácilmente el límite de 180 bytes en eventos con muchos proyectos.

**Flujo de lectura:** El Judge lee `NODUS_BOOTSTRAP_READ` (característica Read) usando `ATT_READ_BLOB_REQ` paginado hasta obtener el payload completo (Decision #36). No se usa `NODUS_DATA_WRITE` para este flujo — esa característica va en dirección Judge→Admin, no Admin→Judge.

**Byte de prefijo:** `0x03` identifica el tipo bootstrap en cada chunk.

| Prefijo | Tipo de payload    |
|:--------|:-------------------|
| `0x01`  | Voto / JSON data   |
| `0x02`  | Media (imagen)     |
| `0x03`  | Bootstrap data     |
| `0x04`  | Project sync delta |
| `0xA1`  | ACK                |

**Documentos afectados:** 04, 10.

## DECISIÓN #28 — Tabla `Judges` en SQLite (Admin y Judge)

**Fecha:** 2026-03-04
**Decisión:** El esquema SQLite incluye una tabla `Judges` explícita con los siguientes campos:

```csharp
class Judge {
    [PrimaryKey] string Id {get;set;}   // UUID generado al registrarse
    string EventId {get;set;}           // FK → Events.Id
    string Name {get;set;}              // Nombre ingresado en Nodus.Judge
    string PublicKeyBase64 {get;set;}   // Clave pública Ed25519 (Base64) — fuente de verdad para verificar firmas

}
```

**Razonamiento:** Sin esta tabla el Admin no puede verificar firmas Ed25519 de los paquetes de voto (la clave pública no tiene dónde vivir), ni puede implementar el Blocklist de jueces (Decisión #8). Es un bloqueador de implementación.
**Documentos afectados:** 03, 04.

## DECISIÓN #29 — Mecanismo Admin→Juez: Pull Periódico por `NODUS_BOOTSTRAP_READ`

**Fecha:** 2026-03-04
**Decisión:** La distribución de nuevos proyectos a jueces ya registrados es **pull-based**:
- El Judge app sondea `NODUS_BOOTSTRAP_READ` cada **`Random(50s, 70s)`** (jitter anticolisión — Decisión #43) durante el evento activo.
- El Admin responde con un paquete `project_sync` (prefijo `0x04`) que contiene solo los proyectos con `SequenceNumber` mayor al cursor del Judge (Decision #39).
- No se requiere una conexión persistente Admin–Juez.

**Para `blocklist_sync`:** El Admin emite el broadcast vía `NODUS_ACK_NOTIFY` a nodos LINK activos (best-effort). El Admin es el punto de enforcement final — un paquete de juez bloqueado que pase por un relay es rechazado igual al llegar al Admin (ver Decisión #8).

**Razonamiento:** Un mecanismo push requeriría conexiones persistentes Admin–Juez (impracticable con Shiny.BluetoothLE). El pull cada 60 s es suficientemente fresco para un evento de horas. Reutiliza la infraestructura GATT existente sin añadir nuevos roles o características.
**Documentos afectados:** 02, 04, 06.

## DECISIÓN #30 — `Project.EventId` como Campo Requerido

**Fecha:** 2026-03-04

## DECISIÓN #34 — "Latest Version Wins": Usar Campo `version`, No `CreatedAtUtc`

**Fecha:** 2026-03-04
**Decisión:** El criterio de desempate para re-evaluaciones es el campo `version` (entero). El registro con mayor `version` gana. `CreatedAtUtc` se conserva únicamente como dato de auditoría.

**Razonamiento:** En un sistema offline, los relojes de los dispositivos no están sincronizados. Dos jueces con relojes desajustados producirían timestamps `CreatedAtUtc` incorrectos que invalidarían las comparaciones. El campo `version` es un contador local monotónico controlado explícitamente por el juez — no depende del reloj del sistema.

## DECISIÓN #35 — `judge_id`: Generado en el Dispositivo del Juez

**Fecha:** 2026-03-04
**Decisión:** El `judge_id` es un **UUID V4 generado en el dispositivo del juez** antes de enviar `judge_register`. Se incluye explícitamente en el paquete `judge_register`:

**Razonamiento:** Si el Admin generara el ID y se lo devolviera al juez, la identidad del juez dependería de recibir una respuesta del servidor antes de poder votar. ## DECISIÓN #36 — `event_bootstrap`: Usa `NODUS_BOOTSTRAP_READ` (Read), No `NODUS_DATA_WRITE`

**Fecha:** 2026-03-04
**Decisión:** El payload `event_bootstrap` se entrega al juez a través de la característica `NODUS_BOOTSTRAP_READ` (tipo GATT: **Read**). El juez emite peticiones `ATT_READ_BLOB_REQ` paginadas de 180 bytes hasta consumir el payload completo.

**La característica `NODUS_DATA_WRITE` (WriteWithoutResponse) va en dirección Judge→Admin exclusivamente.** El Admin no puede "escribir" a un cliente a través de ella — el GATT Write lo inicia el cliente, no el servidor. Usar `NODUS_DATA_WRITE` para bootstrap sería físicamente imposible.

**Flujo correcto:**
1. Judge conecta al Admin via GATT directo.
2. Judge hace `READ_REQ` sobre `NODUS_BOOTSTRAP_READ`.
3. Admin responde con offset 0 (primera página de 180 bytes).
4. Judge repite con `READ_BLOB_REQ(offset=180)`, `READ_BLOB_REQ(offset=360)`, etc.
5. Cuando la respuesta es < 180 bytes, el payload está completo.
6. Judge ensambla el JSON y persiste en SQLite.

**Documentos afectados:** 00 (Dec #27), 04, 10.

## DECISIÓN #37 — `project_sync` No Viaja por el Swarm; Es Pull Directo

**Fecha:** 2026-03-04
**Decisión:** Los paquetes `project_sync` **no se propagan por el Swarm relay**. El mecanismo es pull-based directo:
- El Judge se conecta al Admin (o a un nodo LINK con `NODUS_BOOTSTRAP_READ`) y lee el delta.
- El delta retorna proyectos con `SequenceNumber > since_seq` (ver Decisión #39).
- Los nodos LINK en estado relay **no almacenan ni reenvían** `project_sync`.

**Razonamiento:** Propagar `project_sync` por el Swarm requeriría que los nodos LINK almacenen el payload temporalmente y lo reenvíen — lo cual no está definido en el FSM y rompería la arquitectura de relay stateless. El pull directo reutiliza `NODUS_BOOTSTRAP_READ` sin nuevo protocolo.
**Documentos afectados:** 06.

## DECISIÓN #38 — Admin Dashboard "Unsynced to Cloud": Solo Votos `ReachedAdmin` pendientes de nube

## DECISIÓN #39 — `Project.SequenceNumber`: Cursor de Sincronización Delta

**Fecha:** 2026-03-04
**Decisión:** El modelo `Project` incluye `int SequenceNumber` — un entero auto-incremental **por evento**, asignado por el Admin en el momento de registrar el proyecto.

**Uso en `project_sync`:** El Judge envía `since_seq = [SequenceNumber más alto en su SQLite local]`. El Admin responde con todos los proyectos donde `SequenceNumber > since_seq`. Si el Judge no tiene ningún proyecto, envía `since_seq = 0`.

**El campo `since_version` en el paquete `project_sync` (Decisión #32) se renombra a `since_seq`** para reflejar que es un sequence number, no un contador de versión.

**Razonamiento:** Usar el conteo total de proyectos como cursor es frágil: si el Admin elimina o reordena proyectos, el cursor se desfasa. Un `SequenceNumber` monotónico es el cursor estándar para sync incremental (pattern "event sourcing").
**Documentos afectados:** 02 (project_sync JSON), 03 (Project model).

## DECISIÓN #41 — `blocklist_sync`: Prefijo `0x05` y Formato Canónico

**Fecha:** 2026-03-04
**Decisión:** El paquete `blocklist_sync` usa el prefijo de byte **`0x05`** y se entrega como una GATT Notification en la característica `NODUS_ACK_NOTIFY`. Formato canónico:

```json
{
  "type": "blocklist_sync",

    "BASE64_ED25519_PUBLIC_KEY_1",
```

## DECISIÓN #43 — Jitter en Poll de `project_sync`: Random(50s, 70s)

**Fecha:** 2026-03-04
**Decisión:** El intervalo de polling del Judge a `NODUS_BOOTSTRAP_READ` para `project_sync` es **`Random(50s, 70s)`** en lugar de exactamente 60 s.

**Razonamiento:** 50 jueces registrados en un lapso cercano tendrán sus timers sincronizados. Sin jitter, todos intentan conectarse al Admin GATT simultáneamente cada minuto. GATT soporta ~7-8 conexiones concurrentes; 50 conexiones simultáneas generan un "thundering herd". El jitter ±10 s desincroniza los timers, siguiendo el mismo principio del algoritmo Trickle.
**Documentos afectados:** 02, 06.

## DECISIÓN #44 — `event_bootstrap`: Cifrado con `SharedAESKey`

**Fecha:** 2026-03-04
**Decisión:** El payload `event_bootstrap` se cifra con `SharedAESKey` **antes** de escribirse en el buffer de `NODUS_BOOTSTRAP_READ`. El Judge descifra los chunks ensamblados con la misma clave.

**Razonamiento:** `NODUS_BOOTSTRAP_READ` es una característica BLE legible por cualquier dispositivo que conozca el Service UUID. Sin cifrado, un estudiante con un sniffer BLE obtendría la lista de proyectos y la rúbrica antes de comenzar la votación. El `SharedAESKey` ya existe en el Judge tras el handshake: no hay overhead operacional.
**Documentos afectados:** 04.

## DECISIÓN #45 — Flujo de Cierre de Evento

**Fecha:** 2026-03-04
**Decisión:** Flujo canónico de cierre:

1. Admin presiona **"Close Event"** (diálogo de confirmación requerido).
2. **Efectos inmediatos:**
   - Admin deja de aceptar nuevos `judge_register`.
   - Todos los `Edit Token` de estudiantes quedan inválidos; Nodus.Web retorna `HTTP 410 Gone`.
   - La próxima respuesta de `project_sync` incluye `"event_status": "closed"`.
3. **Grace period (10 min):** Admin sigue aceptando votos BLE entrantes 10 minutos para transmisiones en vuelo.
4. **Post-grace:** Admin rechaza nuevos votos. Judges ven `SyncError` con etiqueta "Event closed".
5. **Export:** Botón "Export Results" se activa al cerrar el evento.

**Razonamiento:** Sin este flujo, `AdminViewModel` y `JudgeViewModel` no tienen respuesta precisa para cuándo termina la responsabilidad de cada actor.
**Documentos afectados:** 06, 07, 13.

## DECISIÓN #50 — GATT Characteristics: Admin vs. LINK Relay (Alcance Diferenciado)

**Fecha:** 2026-03-04
**Decisión:** Admin y nodos LINK comparten el mismo **Service UUID** (`NODUS_MAIN_SERVICE`) para ser reconocibles por los SEEKERs, pero exponen **subconjuntos diferentes de características**:

| Característica | Admin | LINK relay |
|:--------------|:-----:|:----------:|
| `NODUS_DATA_WRITE` | ✅ | ✅ |
| `NODUS_ACK_NOTIFY` | ✅ | ✅ |
| `NODUS_BOOTSTRAP_READ` | ✅ | ❌ |

**`NODUS_BOOTSTRAP_READ` es Admin-only.** Un nodo LINK no posee el catálogo de proyectos ni la rúbrica — intentar exponerla retornaría datos vacíos o incorrectos. Los polls de `project_sync` (Decisión #29) requieren una conexión directa al Admin. Los votos (`NODUS_DATA_WRITE`) sí pueden viajar a través de relays LINK.

**Razonamiento:** Doc 02 §6 (versión anterior) decía erróneamente que Admin y LINK exponen características idénticas, creando la ambigüedad de qué respondería un LINK en un `BOOTSTRAP_READ`. Esta decisión cierra ese gap arquitectónico.
**Documentos afectados:** 02, 09.

## DECISIÓN #51 — Formato Canónico del Vote Payload (Pre-Cifrado)

**Fecha:** 2026-03-04
**Decisión:** El JSON del voto antes de ser cifrado con AES es:

```json
{
  "comment": "optional free-text, max 500 chars, or null",
  "event_id": "EVT-XYZ",
  "judge_id": "UUID-V4-OF-JUDGE",
  "local_audio_path": null,
  "local_photo_path": null,
  "packet_id": "UUID-V4",
  "project_id": "PROJ-A3X",
  "scores": { "c1": 8, "c2": 4.5 },
  "signature": "BASE64_ED25519_SIGNATURE",
  "timestamp_utc": 1741046400,
  "type": "vote",
  "version": 1
}
```

**Regla de firma:**
1. Construir el JSON **sin** el campo `signature`.
2. Serializar a UTF-8 con **keys ordenadas alfabéticamente** (para determinismo multiplataforma).
3. Firmar los bytes UTF-8 con la clave privada Ed25519 del juez.
4. Asignar `signature` como Base64 del resultado.
5. Cifrar el JSON completo (con `signature`) con `SharedAESKey` (AES-256-GCM o AES-256-CBC).
6. Envolver en el envelope de transporte: `{ "packet_id": "...", "hops": [], "payload": "CIPHER_BASE64", "ttl": 2 }`.

**Razonamiento:** Sin este formato definido, dos desarrolladores implementando Judge y Admin producen formatos incompatibles. El ordenamiento de keys garantiza firmas deterministas independientemente del serializador JSON usado.
**Documentos afectados:** 04.

## DECISIÓN #53 — Escala de Batería en Código: `0.0–1.0` (Veto < `0.20`)

**Fecha:** 2026-03-04
**Decisión:** La interfaz `IBleManager.GetBatteryLevel()` retorna un `double` en escala **`0.0–1.0`** (donde `0.15` = 15%). En consecuencia, el veto de batería del FSM Firefly se implementa como:

```csharp
if (bleManager.GetBatteryLevel() < 0.20) return 0; // battery veto
```

**La comparación correcta es `< 0.20`, NO `< 20`.** Usar `< 20` siempre dispararía el veto porque `GetBatteryLevel()` siempre retorna valores ≤ 1.0.

**Razonamiento:** La documentación del spec (Decisión #3) habla de "Battery veto < 20%" en lenguaje humano, pero la interfaz `IBleManager` usa escala 0.0–1.0. Esta inconsistencia lingüística es un bug silencioso clásico: quien lea solo el spec escribe `< 20` y el swarm nunca elige relays. Los tests de Doc 08 usan `Returns(0.15)` correctamente, pero faltaba documentar la razón.
**Documentos afectados:** 08, 12.

## DECISIÓN #54 — "Active Relays": Inferido de los Arrays `hops[]` de Votos Recibidos

**Fecha:** 2026-03-04
**Decisión:** El contador "Active Relays (est.)" del Admin Dashboard **no se basa en un mecanismo heartbeat separado**. Se infiere de la actividad observable:

- **Fuente de datos:** El array `hops[]` de cada paquete de voto recibido + los paquetes `sync_request` (prefijo `0x06`, Decisión #56) recibidos en `NODUS_DATA_WRITE`.
- **Cómputo:** Admin mantiene un diccionario `{judge_id → DateTime LastSeen}`. Cualquier paquete cuyo `hops[]` incluya un `judge_id`, o cuyo emisor directo sea un `judge_id`, actualiza `LastSeen` para ese juez.
- **Definición de "activo":** Un nodo se considera **activo como relay** si ha aparecido en al menos un `hops[]` (es decir, actuó como relay de otro) en los últimos **5 minutos**.
- **"Last Ping"** en el Node Inspector = el `DateTime LastSeen` de ese juez (la última vez que apareció en cualquier paquete, sea como emisor o como relay).

**Ningún protocolo nuevo es necesario.** El dato ya está disponible en el envelope de transporte existente.

**Limitación documentada:** El contador es una **estimación conservadora** — un nodo en estado LINK que no ha relayado ningún voto en los últimos 5 min no aparece en el contador aunque siga activo. Esto es correcto: si no ha relayado nada, no está contribuyendo al swarm en ese momento.

**UI:** La etiqueta en el Sidebar del Dashboard es **"Active Relays (est.): N"** (no "Active Fireflies") para dejar claro que es un valor inferido, no un estado confirmado. El Node Inspector muestra el rol `LINK` o `SEEKER` estimado basado en si ese juez ha aparecido en `hops[]` de otros.

## DECISIÓN #55 — Fórmula de Scoring para Export de Resultados

**Fecha:** 2026-03-04
**Decisión:** El algoritmo de cálculo de puntuaciones para el archivo `Results_Final.xlsx` es:

### Score por juez (por proyecto)

Dado un voto con scores `{c1: s1, c2: s2, ...}` y pesos `{c1: w1, c2: w2, ...}` definidos en la rúbrica:

$$\text{judgeScore} = \frac{\sum_i (s_i \times w_i)}{\sum_i w_i}$$

Esto produce un score **normalizado al rango de la rúbrica** (independiente del número de criterios y sus pesos).

### Average Score del proyecto

$$\text{AverageScore} = \frac{1}{N_{judges}} \sum_{j=1}^{N_{judges}} \text{judgeScore}_j$$

Media aritmética de los scores por juez. Solo se usa el voto con **mayor `version`** por par (judge_id, project_id) — Latest Version Wins (Decisión #7).

### Total Points del proyecto

$$\text{TotalPoints} = \sum_{j=1}^{N_{judges}} \sum_i (s_{ji} \times w_i)$$

Suma bruta de todos los scores ponderados de todos los jueces. **NO está normalizada.** Sirve como criterio de desempate cuando dos proyectos tienen el mismo `AverageScore` redondeado — el que acumuló más `TotalPoints` gana.

### Implementación .NET (referencia)

```csharp
double CalcJudgeScore(Vote v, RubricCriteria[] criteria)
{
    double weightedSum = criteria.Sum(c => (double)v.Scores[c.Id] * c.Weight);
    double totalWeight  = criteria.Sum(c => c.Weight);
    return weightedSum / totalWeight;
}

double CalcProjectAverage(IEnumerable<Vote> latestVotes, RubricCriteria[] criteria)
    => latestVotes.Average(v => CalcJudgeScore(v, criteria));

double CalcTotalPoints(IEnumerable<Vote> latestVotes, RubricCriteria[] criteria)
    => latestVotes.Sum(v => criteria.Sum(c => (double)v.Scores[c.Id] * c.Weight));
```

**Razonamiento:** Un score no ponderado (`Σ s_i / N`) ignora los pesos definidos en la rúbrica, haciendo que criterios de peso `2.0` valgan igual que criterios de peso `0.5`. El score ponderado normalizado es la métrica estándar para evaluaciones multi-criterio y es coherente con el campo `weight` del rubric JSON (Decisión #40). `TotalPoints` no normalizado permite ordenar proyectos con scores medios iguales pero diferente número de votos (más jueces = más puntos = desempate justo).

## DECISIÓN #56 — `sync_request`: Protocolo para Pasar `since_seq` en Poll de `project_sync`

**Fecha:** 2026-03-04
**Decisión:** La característica GATT `NODUS_BOOTSTRAP_READ` (tipo: Read) no permite al cliente incluir parámetros en la request (`ATT_READ_BLOB_REQ` no tiene cuerpo). Para que el Admin sepa qué `since_seq` usar al construir la respuesta `project_sync`, el Judge realiza el siguiente flujo dentro de la misma conexión GATT:

**Flujo canónico de poll `project_sync`:**

1. Judge conecta al Admin GATT Server (si no está ya conectado).
2. Judge escribe en `NODUS_DATA_WRITE` (WriteWithoutResponse):
   ```json
   {
     "type": "sync_request",
     "judge_id": "UUID-V4-OF-JUDGE",
     "since_seq": 42
   }
   ```
   Prefijo de byte: **`0x06`** (nuevo — ver tabla de prefijos en Doc 02 §4).
3. Admin recibe el `sync_request`, computa el delta `project_sync` filtrando `WHERE SequenceNumber > 42`, y almacena el resultado en un buffer **per-judge** (clave: `judge_id`).
4. Judge lee `NODUS_BOOTSTRAP_READ` usando `ATT_READ_BLOB_REQ` paginado (mismo protocolo que `event_bootstrap`, Decisión #36).
5. Admin sirve el buffer pre-computado para ese `judge_id`. El buffer se elimina tras completarse la lectura o al caer la conexión.

**Estado en Admin:** El Admin mantiene un `Dictionary<string, byte[]>` (`judge_id → payload serializado`) en memoria, con TTL de **30 segundos**. Si el Judge no lee el resultado en 30 s, el buffer se descarta (el Judge puede reintentar).

**Caso bootstrap inicial (primer registro):** El paquete `judge_register` (Decisión #35) implícitamente tiene `since_seq = 0`. El Admin prepara el `event_bootstrap` completo (prefijo `0x03`) como respuesta. No es necesario enviar un `sync_request` separado para el bootstrap inicial.

**Backward compatibility:** Si el Admin recibe una lectura `NODUS_BOOTSTRAP_READ` sin un `sync_request` previo de ese `judge_id` (por ejemplo, un cliente antiguo), responde con el `event_bootstrap` completo (todos los proyectos, `since_seq = 0`). Esto evita que el sistema rompa si hay una desconexión entre el paso 2 y el paso 4.

**Razonamiento:** `ATT_READ_BLOB_REQ` no soporta parámetros. Las alternativas eran (A) este mecanismo write-then-read, (B) retornar siempre todos los proyectos sin filtrar, o (C) añadir una quinta característica GATT. La opción A es la más correcta para eventos con >50 proyectos: el Admin solo transmite el delta real. La opción B funcionaría en eventos pequeños pero degrada a medida que crecen los proyectos. La opción C añade superficie API innecesaria.
