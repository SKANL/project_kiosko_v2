# 07. Logistics: Event Runbook

> **Status:** Draft
> **Audience:** Event Organizers / Tech Lead

## 1. Hardware Checklist (The Survival Kit)

### The Command Center

- [ ] **1x Windows Laptop:** Core i7, 16GB RAM minimum.
- [ ] **1x USB Bluetooth Dongle:** Baseus or TP-Link (Bluetooth 5.0+). Do NOT rely on internal chipsets.
- [ ] **1x UPS (No-Break):** The server MUST NOT lose power.
- [ ] **1x Wi-Fi Router:** For the Laptop's local network.
- [ ] **1x Projector/Screen:** CRITICAL for displaying the "Registration QR".

### Spare / Backup Judge Devices (Optional but Recommended)

- [ ] **4x Android Phones:** Pixel 6 or Galaxy S21+ equivalent.
- [ ] **4x Power Banks:** To keep phones charged for 8 hours.

> **Important:** These phones have **no special configuration** as bridges. The Firefly Swarm protocol automatically elects any judge device as a relay node based on signal quality. Bringing extra phones simply increases the density of available relay candidates, improving swarm resilience. Do NOT attempt to manually configure any phone as a fixed bridge — that role does not exist in the protocol.

## 2. Software Installation (The Headache)

_Constraint:_ There is no Internet. We cannot use the App Store on the day.

### Nodus.Admin (Windows Laptop)

- **Install:** Run the `NodusAdmin_Setup.exe` installer. It installs the app and the embedded Nodus.Web static files into `{AppBase}/Web/`.
- **Firewall (CRITICAL):** The installer adds a Windows Firewall inbound rule for **TCP port 5000**. Verify it is active:
  ```
  netsh advfirewall firewall show rule name="NodusAdmin HTTP"
  ```
  If the rule is missing, add it manually:
  ```
  netsh advfirewall firewall add rule name="NodusAdmin HTTP" protocol=TCP dir=in localport=5000 action=allow
  ```
  Without this rule, students on the venue Wi-Fi cannot open the registration page. (Decision #52)

### Android

- **Method:** Sideloading via USB OTG.
- **Prep:** Have the `.apk` file on 5 USB drives (USB-C).
- **Process:** Plug -> Install -> Warn about "Unknown Sources" -> Accept.

### iOS (The Danger Zone)

- **Method:** TestFlight Public Link (Must happen 24h before).
- **Pre-Event:** Email all judges the TestFlight link. They MUST install at home.
- **On-Site:** If they didn't install it, **THEY CANNOT BE JUDGES**. Have backup Android tablets ready for them.

## 3. Disaster Recovery

### Scenario A: Server Laptop Crashes/Updates

1.  **Don't Panic:** The Judges' phones have the "Source of Truth" in their SQLite `Pending` rows.
2.  **Restart:** Reboot Laptop → Open **Nodus.Admin**.
3.  **Recover:** The Server DB might be empty/corrupt?
    - _Action:_ Restore from last backup (auto-backup every 5 mins).
      > **Backup Mechanism (Decision #48):** Nodus.Admin runs a background timer that every **5 minutes** executes an atomic `File.Copy()` of the SQLite database file to `{AppData}\Nodus\Backups\nodus_{timestamp}.db`. Only the **3 most recent** backup files are retained; older files are deleted automatically. The timestamp format is `yyyyMMdd_HHmmss`.
      >
      > **Restore procedure:** From the "Database" menu in Nodus.Admin, click **"Restore from Backup"** → select a `.db` file from the `Backups/` folder → confirm. The app restarts automatically after restore.
    - _Resync:_ Judges' phones will automatically re-transmit all votes with `ScoreStatus = Pending`. **Data loss window is bounded by the backup interval (5 min max).**
    - _Note:_ Votes already confirmed (`ScoreStatus = ReachedAdmin`) will **not** be re-transmitted by the Judge — the app considers them delivered. Those votes are only recoverable from the last backup. This is why the auto-backup is non-negotiable.

### Scenario B: 2.4GHz jam (Microwaves/Too many people)

1.  **Pivot:** Switch to "Mule Mode" (Proximity Sync).
2.  **Instruction:** Announce on PA: "Judges, please come to the Command Table to sync your votes manually."
3.  **Action:** Judges walk to within 1 meter of the Server Dongle.
4.  **Verification:**
    - **Judge's Phone:** Ensure Bluetooth is ON. Watch for the Status Indicator in the TopAppBar.
    - **Scores:** As soon as the Judge is within BLE range of the Admin (RSSI > -80 dBm), all `ScoreStatus = Pending` votes transmit automatically via the Swarm. The TopAppBar turns **Green** (☁️).
    - **Media:** If additionally within ~1 metre (RSSI > -60 dBm), media files (photos/audio) transfer via BLE chunked sync. A Toast Notification **"Media Synced!"** appears when complete.
    - **Server:** Verify the `Media/{EventId}` folder is populating with `.jpg` files.
5.  **Troubleshooting:**
    - If sync doesn't start, toggle Bluetooth OFF/ON on the Judge's phone.
    - Ensure the Judge is _physically close_ (RSSI > -60dBm is mandatory).

## 4. End of Event Procedure (Closing)

Follow this sequence to close the event cleanly (Decision #45).

### Step 1 — Verify Coverage
Before closing, check the Admin Dashboard Grid:
- No projects with `🛑 No Data` (zero votes). Alert via PA if any.
- Sidebar **"Unsynced to Cloud"** counter is zero, OR accept that cloud sync will happen later.

### Step 2 — Close the Event
1.  In Nodus.Admin, press **"Close Event"** and confirm the dialog.
2.  The system immediately:
    - Stops accepting new `judge_register` packets.
    - Invalidates all student Edit Tokens (Nodus.Web returns `HTTP 410 Gone` for `/edit?token=...`).
    - Marks the event as closed in SQLite.
3.  Announce over PA: _"Voting is now closed. Please submit any last scores within the next 10 minutes."_

### Step 3 — Grace Period (10 min)
Do NOT power off the laptop. The Admin accepts pending in-flight votes for **10 more minutes**. Watch the "Total Votes" counter on the dashboard — it should stabilize.

### Step 4 — Export Results
1.  Click **"Export Results"** → opens a Save dialog → saves `Results_Final.xlsx`.
2.  Copy the file to a USB drive for the awards ceremony projection.
3.  _(Optional)_ If internet is available: Click **"Sync to Cloud"** → pushes all data to MongoDB via Nodus.API.

### Step 5 — Shutdown
1.  Close Nodus.Admin.
2.  Power off the BLE dongle last (to avoid orphaned connections).
