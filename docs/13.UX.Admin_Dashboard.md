# 13. UX: Admin Dashboard (The Control Tower)

> **Status:** Draft
> **Platform:** Windows Desktop (MAUI / WinUI)
> **Role:** The Central Nervous System

## 1. Pre-Event Setup Flow (At Home/Office)

The Admin must configure the event _before_ going offline.

### A. Event Setup & QR Generation

The Admin generates exactly **3 QR codes** during event setup (canonical taxonomy defined in Doc 04):

1.  **Event Registration QR** (for Students):
    - Action: Click "Enable Registration".
    - Content: `http://[LOCAL_IP]:5000/register?event=EVT-ID` (local network URL — no internet required).
    - Display: Project this on the wall. Students scan with their camera app.

2.  **Judge Access QR** (for Judges):
    - Action: Click "Generate Judge QR" → Set Event Password.
    - Content: `AES_Encrypt({SharedAESKey, ServerUUID, EventID}, PBKDF2(Password, EventId))` (Decision #22).
    - Display: Show on projector. Judges scan with Nodus.Judge app.

3.  **Project Display QR** (per project, auto-generated):
    - Generated automatically when a project is registered.
    - Content: `nodus://vote?pid=PROJ-A3X`.
    - Displayed on the student's phone/printout at their stand.

> There is no separate "MASTER QR", "Security QR", or "Admin QR". The Judge Access QR is the single authentication mechanism for judges.

## 2. Live Monitoring (The Dashboard)

During the event, this is the "Flight Control" screen.

### A. Live Registration Monitor

- **View:** Real-time list of registering students.
- **Action:** Approve/Reject/Edit.
- **Manual:** "+" button for walk-ins.

### B. The "Pulse" (Sidebar)

- **Total Votes:** `1,240` (Big Counter).
- **Active Relays (est.):** `12` (Judges inferred to be acting as BLE relay nodes, based on appearing in `hops[]` arrays of votes received in the last 5 minutes — Decision #54. This is an estimate: a relay that hasn’t forwarded any vote recently won’t appear here even if its BLE is still active.)
- **Unsynced to Cloud:** `45` (Votes with `ScoreStatus = ReachedAdmin` that have not yet reached MongoDB — pendientes de sincronización a la nube via Nodus.API. Decision #38).

### C. The Grid (Main View)

A list of all Projects being evaluated.
| ID | Project Name | Votes / Judges | Status | Last Activity |
|:---|:-------------|:--------------|:-------|:--------------|
| 01 | Team Alpha | 5/5 | ✅ Done | 2 min ago |
| 02 | Team Beta | 2/5 | ⚠️ Low | 45 min ago |
| 03 | Team Gamma | 0/5 | 🛑 No Data | NEVER |

- **Alerts:** Rows turn Red if no votes received in > 1 hour.
- **Denominator:** The `/N` is dynamic — equals the number of judges currently registered in the event (Decision #21). It increments automatically when each new judge completes onboarding. The Node Inspector shows which specific judges have not yet voted on a given project.

### D. The Node Inspector (Troubleshooting)

Shows a list of Judges from which Admin has received recent activity.

- **Judge:** Alice
- **Battery:** 85%
- **Role (est.):** LINK (appeared in `hops[]` of other judges’ votes) 🟢
- **Last Seen:** 2s ago (timestamp of last packet received from or through this judge — Decision #54)

> The Node Inspector shows **estimated** roles. `LINK` means this judge has relayed at least one packet in the last 5 minutes. `SEEKER` means we’ve received their own votes but haven’t seen them in any `hops[]`. A judge not appearing in the list has sent no data yet.

## 3. End of Event (Closing)

### A. Pre-requisites Before Closing

- Verify the **Grid** shows no rows with `Status = 🛑 No Data`. Alert the PA if any project has zero votes.
- Confirm **Unsynced to Cloud** counter (sidebar) is zero or Wi-Fi/Ethernet is available for a final cloud push.

### B. Closing the Event

1.  **Action:** Admin clicks **"Close Event"** button (confirmation dialog required).
2.  **Immediate effects:**
    - Nodus.Admin stops accepting new `judge_register` packets.
    - All `Edit Token`s for student registrations are invalidated. Nodus.Web returns `HTTP 410 Gone` for any `/edit?token=` request (Decision #45).
    - The next `project_sync` poll response to every Judge includes `"event_status": "closed"`.
3.  **Grace period (10 min):** Admin continues accepting incoming vote packets for 10 minutes to allow in-flight BLE transmissions to complete (Decision #45).
4.  **After grace period:** Admin rejects new vote packets. Dashboard shows a **"CLOSED"** badge on all active judge rows.

### C. Rubric Lock Policy

> **Rule (Decision #46):** The evaluation rubric (`Event.EvaluationRubricJson`) is **immutable once the first vote exists in the database**. Attempting to edit the rubric after that point returns an error in the Admin UI: _"Rubric cannot be changed after voting has started."_

This prevents the situation where some judges voted on rubric v1 and others on rubric v2, making scores incomparable.

## 4. Data Export (Awards Ceremony)

1.  **Export Excel:** Click **"Export Results"** → Generates `Results_Final.xlsx` immediately from local SQLite. No internet required. Available as soon as the event is closed.
2.  **Cloud Sync:** (If Ethernet/internet available) Click **"Sync to Cloud"** → pushes all event data to MongoDB via `Nodus.API`. Endpoint: `POST /api/sync/event`.
