# 06. UX: User Flows & Optimistic UI

> **Status:** Draft
> **Key Principle:** "Don't Make Me Wait"

## 1. The "Optimistic" Philosophy

In Nodus, network latency can be 5 seconds or 5 hours. The UI **must never block** the user while waiting for a connection.

- **Wrong:** User taps "Submit" -> Spinner appears -> Waits 10s -> "Error".
- **Right:** User taps "Submit" -> Animation plays -> Form closes -> "Saved!" toast appears.

## 2. Visual Feedback States (The Traffic Light)

We use a persistent **Status Indicator** in the TopAppBar.

| Icon | Color | State       | Meaning                                                                              |
| :--- | :---- | :---------- | :----------------------------------------------------------------------------------- |
| ☁️   | Green | **Synced**  | Scores reached the Admin node (`ScoreStatus = ReachedAdmin`). Full cloud sync may still be in progress. |
| 💾   | Amber | **Pending** | Votes saved on device, BLE swarm not yet reachable (`ScoreStatus = Pending`).       |
| ⚠️   | Red   | **Risk**    | `ScoreStatus = SyncError` — or Bluetooth radio disabled / storage full.             |

## 3. Core User Flows

### A. The "Judge Onboarding" Flow (Scan & Go)

Distinction: Judges use the **Nodus.Judge app (mobile)**. Students use the **Nodus.Web (browser)**.

1.  **Start:** Judge opens Nodus.Judge app.
2.  **Scan:** Taps "Join Event" → Scans the **Judge Access QR** (projected on screen by Admin).
3.  **Authentication:**
    - App prompt: "Enter Event Password".
    - User types the password announced orally by the organizer (e.g., "Tesla2026").
    - _Logic:_ App decrypts the QR payload by deriving the AES key with `PBKDF2(password, EventId)` (Decision #22).
      - Success: Decryption produces `{SharedAESKey, ServerUUID, EventID}`. "Access Granted".
      - Fail: "Invalid Password" — the QR is cryptographically useless without the correct password.
4.  **Identity (On-Device Key Generation):**
    - App asks: "What is your name?" (e.g., "Dr. Brown").
    - App generates an `Ed25519` key pair locally — **no internet required**.
    - App generates a UUID V4 as the `judge_id` — this is the permanent identifier used in all `Vote` records this judge creates (Decision #35).
    - A `judge_register` packet (containing `judge_id`, name, and public key, encrypted with `SharedAESKey`) is sent to the Admin node via direct GATT connection.
5.  **Bootstrap + Feedback:**
    - _Immediate:_ "Welcome to Hackathon 2026, Dr. Brown".
    - _Bootstrap:_ Admin responds with `event_bootstrap` — the current project list and rubric. App saves them to local SQLite. The judge can now vote fully offline.
    - _Background:_ App stores keys + server UUID. BLE Swarm loop starts. Projects registered **after** onboarding are fetched via periodic polling every `Random(50s, 70s)` (jittered — Decision #43). The poll protocol is two-step: (1) Judge writes a `sync_request` packet (prefix `0x06`) to `NODUS_DATA_WRITE` containing its highest local `SequenceNumber` as `since_seq`; (2) Judge reads `NODUS_BOOTSTRAP_READ` to receive the delta of newer projects. Admin returns only projects with `SequenceNumber > since_seq` (Decision #29, #37, #39, #56). New projects are merged into local SQLite automatically.

### B. The "Voting" Flow (The Happy Path)

1.  **Trigger:** Judge scans a Project QR (e.g., "Team Alpha").
2.  **Load:** App loads schematic from Local SQLite (Instant).
3.  **Input:** Judge manipulates Sliders (1-10) and types comment.
4.  **Media (Optional):** Tap 📷 to attach a photo or 🎤 to record a short voice note as evidence. Both are optional — skipping them does not block submission (Decision #20).
5.  **Submit:**
    - Judge taps "Send Vote".
    - UI: Button turns Green with Checkmark.
    - Navigation: Popped back to Scanner immediately.
    - _Logic:_ Vote inserted to SQLite (scores + any optional media paths). Background Worker wakes up.

### C. The "Swarm" Feedback (Invisible)

The user **never** sees "Connecting to Bridge Node...". Use vague but reassuring language.

- _Notification:_ "Sending votes via nearby mesh..." (If stuck in Amber state for > 5 mins).

### D. The "LINK" State (Positive Reinforcement)

When the Firefly Protocol promotes a device to a **LINK** (Relay) node:

- **Visual:** A small "Glow" or "Firefly Icon" appears in the Status Bar.
- **Toast:** "You are helping 3 nearby judges connect! Keep the app open."
- **Why?** Encourages the user (especially on iOS) to keep the screen active, which ensures the relay stays alive.

## 4. Error Handling

- **Duplicate Scan:** If Judge scans a project they already evaluated → Show **"Re-evaluate"** screen (pre-loaded with previous scores). **Internally always creates a new Vote record** with `version` incremented. Server applies "Latest Version Wins" rule (by `version` field — higher version number wins; Decision #34). The screen title reads "Update Evaluation" to avoid confusion with editing.
- **Bluetooth Off:** Aggressive Alert. "Nodus requires Bluetooth. Enable now?"

### E. The "Mule Mode" (Fail-Safe)

**Trigger:** App is in Mule Mode (no BLE signal from Server or any LINK node for > 10 min, Decision #23) and has `Pending` votes.

1.  **Alert:** Status Bar turns **PURPLE** (Mule Mode).
2.  **Notification:** "Network Unreachable. Data saved locally."
3.  **Instruction:** "Walk to the Admin Desk to sync."
4.  **Auto-Recovery (two-stage):**
    - **Stage 1 — Scores (RSSI > -80 dBm):** As soon as any `NODUS_MAIN_SERVICE` advertisement is detected (Admin or LINK node), Mule Mode exits. All `ScoreStatus = Pending` votes transmit automatically via the Swarm. Status Bar turns **Green** (☁️). The judge does not need to be standing at the desk for this stage.
    - **Stage 2 — Media (RSSI > -60 dBm, proximity required):** If additionally the Admin node is within ~1 metre, the BLE chunked media transfer activates automatically (Decision #23, Doc 10). **Toast:** "🖼️ Media Synced! Thanks for walking."

### F. The "Event Closed" State

**Trigger:** On the next `project_sync` poll, the Admin response carries `"event_status": "closed"` (Decision #45).

1.  **Banner:** A permanent amber banner appears at the top of the app: "⏹️ Voting has ended. Finalise any remaining scores."
2.  **Grace window (10 min):** Judges can continue submitting and syncing votes during a **10-minute grace period** after the event is closed. The Admin accepts incoming vote packets until the window expires.
3.  **After grace period:** The Admin silently rejects new vote packets. The local SQLite write still succeeds (no data loss). The status indicator shows a permanent `SyncError` with label _"Event closed — votes not accepted."_
4.  **Polling stops:** The app stops polling `NODUS_BOOTSTRAP_READ` for `project_sync` deltas.
5.  **Re-evaluate disabled:** The "Re-evaluate" screen becomes read-only, showing a note: _"Voting period has ended."_
