## Resumen Ejecutivo del Problema

La documentación describe un sistema valioso e innovador, pero presenta **inconsistencias críticas** que, si no se resuelven antes de continuar el desarrollo, producirán código difícil de mantener e incluso funcionalidades que no pueden coexistir. Este documento cataloga cada problema, lo ubica en los documentos fuente y propone opciones de resolución para debate.

## INCONSISTENCIA #1 — La Aplicación del Admin: ¿Escritorio o Móvil?

### El Conflicto

La documentación define el **Admin Node** de dos maneras incompatibles:

| Documento | Lo que dice |
|:----------|:------------|
| **01** §4A | "Device: Windows Laptop / Tablet" |
| **05** §1 | Target platforms: Android, iOS, **Windows** |
| **13** §Header | "Platform: Windows Desktop (MAUI / WinUI)" |
| **Requisito del usuario** | "una app móvil para que la persona a cargo cree y gestione el evento" |

### Impacto

El código existente (`Nodus.Server`) está construido como una **aplicación Windows Desktop con WinUI 3**. Si el admin necesita una app móvil, se requiere duplicar o refactorizar completamente la app de administrador.

### Opciones a Debatir

**Opción A — Mantener Windows como admin (recomendado para MVP)**
- El organizador del evento usa un laptop en la "mesa de comando".
- Pragmático: el laptop puede ser el servidor BLE permanente (no entra en cooldown).
- Riesgo: el organizador queda atado a una mesa.

**Opción B — App Móvil para Admin (requisito del usuario)**
- El organizador usa un teléfono Android/iOS.
- Requiere reimplementar `Nodus.Server` como `Nodus.Admin` en MAUI.
- Problema técnico: GATT Server en Android/iOS tiene limitaciones de concurrencia menores que en Windows. El teléfono del admin ahora tiene restricciones de iOS como cualquier Judge.

**Opción C — Admin es Mobile + Nodus.Server es un proceso headless**
- El proceso servidor BLE corre en un mini-PC o Raspberry Pi sin pantalla.
- El Admin móvil se conecta a ese servidor por Wi-Fi local para control.
- Mayor complejidad, pero más flexible en campo.

> **Decisión requerida:** ¿Qué plataforma usa el organizador del evento?

## INCONSISTENCIA #3 — FSM: Dos Implementaciones con Parámetros Distintos

### El Conflicto

El mismo FSM "Firefly" está definido en **dos documentos con parámetros contradictorios**:

| Parámetro | Doc 02 §3 | Doc 12 §3B |
|:----------|:----------|:-----------|
| Random wait (Trickle) | `Random(2s, 10s)` | `Random(5s, 30s)` |
| Threshold RSSI → CANDIDATE | `> -75dBm` | `> -75dBm` (§3B) / `> -70dBm` (código §5) |
| Nombre de los estados | SEEKER, CANDIDATE, LINK, COOLDOWN | SEEKER, CANDIDATE, LINK, COOLDOWN |

Y en **Doc 09**, el Action Plan referencia un FSM con nombres completamente distintos:
> "Replace the simple `IsScanning` boolean with the full Hero State Machine (**IDLE, LOST, HERO, TIRED**)"

### Impacto

Tres conjuntos de nombres para los mismos estados. Developers leyendo docs diferentes implementarán FSMs incompatibles.

### Resolución Propuesta

Consolidar en **un único documento canónico** (Doc 02 o una nueva versión). Eliminar la sección de FSM de Doc 12 y referenciar Doc 02. Eliminar la nomenclatura IDLE/LOST/HERO/TIRED de Doc 09.

**Parámetros canónicos propuestos (para decidir):**

| Parámetro | Propuesta |
|:----------|:----------|
| Threshold RSSI → CANDIDATE | `-75 dBm` |
| Random wait | `Random(3s, 15s)` — balance entre 2/10 y 5/30 |
| Max Link duration | `60s` (consenso en ambos docs) |
| Cooldown duration | `5 min` (consenso) |

## INCONSISTENCIA #4 — Max Hops: TTL=2 vs Diagrama de 3 Saltos

### El Conflicto

| Documento | Lo que dice |
|:----------|:------------|
| **Doc 02** §4 | `"ttl": 2 // Max Hops hardcoded to 2 to prevent loops` |
| **Doc 12** §3C | Diagrama: Node C → B → A → Server = **3 saltos** |

Un TTL=2 significa que el paquete puede pasar por **2 nodos relay antes del servidor**, resultando en una cadena de 3 saltos (Judge → Link1 → Link2 → Server). Pero el texto de Doc 02 dice "Max Hops hardcoded to 2", lo cual implicaría solo 1 relay (Judge → Link → Server).

### Impacto

Si el código implementa `ttl=2` como "máximo 2 nodos en el `hops` array", en realidad son 3 saltos físicos, que es lo que muestra Doc 12. Esto es un problema de definición: ¿TTL cuenta el origen o solo los relays?

### Resolución Propuesta

Redefinir explícitamente:
- `ttl` = número de **relays intermedios permitidos** (no contando origen ni destino).
- `ttl=2` → Judge → Relay1 → Relay2 → Server. **Esto es 2 relays y es consistente con el diagrama de Doc 12.**
- Actualizar el comentario en el JSON de ejemplo para eliminar la ambigüedad.

## INCONSISTENCIA #5 — Transmisión de Media: BLE vs Wi-Fi (Contradicción Directa)

### El Conflicto

| Documento | Lo que dice |
|:----------|:------------|
| **Doc 03** §3B | "Photos and Voice Notes are **NEVER** sent via Bluetooth." |
| **Doc 10** §1 | Define un protocolo BLE completo para transferir imágenes por Bluetooth con chunking, ACKs, y `MediaSyncService`. |

### Impacto

Esta es una **contradicción arquitectónica directa**. O las fotos van por BLE o no van. El código en Doc 10 existe e implica que sí van por BLE en "Mule Mode" (acercándose al servidor). Doc 03 dice que esperan Wi-Fi.

### Opciones a Debatir

**Opción A — Solo Wi-Fi para media (Doc 03 es canónico)**
- Eliminar `MediaSyncService` BLE de Doc 10 y del código.
- Las fotos solo suben cuando hay Wi-Fi real o conexión física.
- Más simple. Menos datos por BLE = mejor rendimiento del swarm.

**Opción B — BLE para media solo en Mule Mode (Doc 10 es canónico)**
- Cuando el Judge se acerca físicamente al servidor (RSSI > -60 dBm), las fotos se transfieren por BLE.
- Se justifica porque en "Mule Mode" la distancia es tan corta que BLE tiene suficiente ancho de banda.
- Más complejo pero no requiere Wi-Fi en ningún momento.

> **Recomendación:** Opción B si el evento puede carecer completamente de Wi-Fi. Opción A si hay siempre una red local disponible.

## INCONSISTENCIA #6 — Formato del Project ID: Cuatro Versiones

### El Conflicto

El ID de proyecto aparece con **4 formatos distintos** en la documentación:

| Documento | Formato de ID | Ejemplo |
|:----------|:-------------|:--------|
| **Doc 03** | `"PROJ-xyz"` | `PROJ-abc` |
| **Doc 04** | Free-form | `PROJECT_004` |
| **Doc 14** | Sequential o Hash | `P-001` |
| **Doc 15** | Sequential con prefix | `PROJ-023` |

Y en los QR payloads:
- Doc 04: `nodus://vote?pid=PROJECT_004&cat=INNOVATION`
- Doc 14: `nodus://vote?pid=P-001&cat=Soft`
- Doc 15: `pid=PROJ-023`

### Impacto

El QR que el juez escanea debe decodificarse en el cliente. Si los IDs tienen formatos distintos, el parser puede fallar. Es un bug latente.

### Resolución Propuesta

Definir **un único formato canónico**:
```
PROJ-{3_CHARS_ALFANUM_MAYUSCULA}
Ejemplo: PROJ-A3X, PROJ-B7K
```
QR payload canónico: `nodus://vote?pid=PROJ-A3X&cat=SOFTWARE`

## INCONSISTENCIA #7 — Estrategia de "Edición de Voto": Append-Only vs Edit Screen

### El Conflicto

| Documento | Lo que dice |
|:----------|:------------|
| **Doc 03** §4 | "We typically do **not allow editing** votes... We create a **new** Vote record (Version 2)." |
| **Doc 06** §4 | "Duplicate Scan: If Judge scans 'Team Alpha' again → Show **'Edit Vote' screen**, not a new blank form." |

### Impacto

La base de datos implementa append-only (versiones), pero la UX muestra una pantalla de "edición". Esto es confuso para el desarrollador: ¿la pantalla de edición crea un nuevo registro o modifica el existente?

### Resolución Propuesta

Unificar el lenguaje: la pantalla se llama "Re-evaluate" o "Update Evaluation". Internamente **siempre crea un nuevo registro** con `version: 2`. La UI muestra los valores previos pre-cargados para conveniencia. El servidor aplica "Latest Version Wins".

## INCONSISTENCIA #8 — Seguridad: Pre-registro Requerido vs. On-the-fly

### El Conflicto

| Documento | Lo que dice |
|:----------|:------------|
| **Doc 04** §1A | "Before the event (at home), the Judge **logs in normally via Internet**. App generates Ed25519 key pair. Public Key sent to Server." |
| **Doc 06** §3A | Judge flow: scan QR → enter password → app asks "What is your name?" → **genera claves localmente**. Sin pre-registro previo. |

### Impacto

- Si se requiere pre-registro online: Judges que no se pre-registraron **no pueden participar**.
- Si las claves se generan on-the-fly: el servidor no tiene la clave pública previamente, por lo que **no puede verificar firmas** al recibir el primer paquete.

Esta inconsistencia afecta directamente al modelo de seguridad anti-cheat.

### Opciones a Debatir

**Opción A — Pre-registro obligatorio (Doc 04)**
- Más seguro. El servidor ya conoce al juez.
- Operacionalmente riesgoso: ¿qué pasa con el juez de último minuto?
- Requiere flujo online previo al evento.

**Opción B — Bootstrap offline con PKI (recomendado)**
- El juez genera claves on-the-fly (Doc 06).
- El **primer paquete** del juez incluye su clave pública firmada con la `SharedAESKey` del evento.
- El servidor la acepta como "trust-on-first-use" y la registra.
- Simple, funciona offline, pero requiere que la `SharedAESKey` sea el mecanismo de autenticación primario.

## INCONSISTENCIA #9 — App del Estudiante: Web PWA vs BLE

### El Conflicto / Gap de Requisitos

El usuario quiere usar **BLE de los celulares** para el sistema. Pero la app del estudiante es una **Web PWA (Blazor)**. Los navegadores web tienen soporte **muy limitado** de Web Bluetooth:
- Chrome en Android: soporte parcial.
- Safari en iOS: **Web Bluetooth no soportado**.
- Firefox: **Web Bluetooth no soportado**.

La app del estudiante (Web) **no puede participar en el swarm BLE**. Solo se comunica vía HTTP/LocalStorage.

### Impacto Arquitectónico

Esto crea **dos capas de transporte separadas** en el mismo evento:
1. **BLE Swarm** → Jueces ↔ Admin (funciona offline)
2. **HTTP / Wi-Fi local** → Estudiantes ↔ Admin (requiere red local)

El servidor admin necesita servir ambos protocolos simultáneamente.

### Opciones a Debatir

**Opción A — Mantener Web para estudiantes (simplicidad)**
- Los estudiantes no necesitan BLE: solo registran su proyecto y muestran un QR.
- El server admin expone un endpoint HTTP local (ej. `http://192.168.1.1:5000/register`).
- Requiere que el router Wi-Fi local esté funcionando.
- Es la opción más viable y simple.

**Opción B — App nativa MAUI para estudiantes**
- Los estudiantes instalan la misma app que los jueces (roles diferenciados).
- Permite que los estudiantes participen en el swarm BLE.
- Problema: fricción de instalación en el evento (sideloading para no-jueces).
- La web es más accesible (solo abrir URL).

> **Recomendación fuerte: Opción A.** La web es correcta para estudiantes. El BLE se reserva para jueces y admin. Pero esto debe documentarse explícitamente como una decisión arquitectónica.

## INCONSISTENCIA #10 — Registro de Estudiantes con Red o sin Red

### El Conflicto

Doc 15 dice que el estudiante abre `nodus.app/register?event=XYZ` escaneando un QR. Pero Doc 01 y Doc 07 dicen que hay problemas de internet en el venue.

- Si `nodus.app` es un dominio público: requiere internet.
- Si es una red local: la URL debería ser `http://192.168.1.x:port/register`.

Doc 07 menciona que se lleva **un router Wi-Fi local**, pero nunca define si Nodus.Web corre localmente o en la nube.

### Impacto

Si Nodus.Web corre solo en la nube (Supabase/hosting externo), el registro de estudiantes **falla completamente** sin internet, que es exactamente el escenario que el sistema promete resolver.

### Resolución Propuesta

Definir explícitamente que Nodus.Web corre en modo dual:
1. **Modo Evento (Local):** Nodus.Web corre en el mismo proceso que Nodus.Server (o como un proceso adjunto) en el laptop admin. URL: `http://[IP-local]:5000`.
2. **Modo Cloud (Backup):** Si hay internet, puede sincronizar con Supabase.

El QR de registro debe contener la **IP local del servidor**, no un dominio público.

## INCONSISTENCIA #11 — Runbook Menciona "Bridges Fijos" que Contradicen el Protocolo Dinámico

### El Conflicto

Doc 07 §1 (Hardware Checklist):
> "4x High-End Android Phones: Pixel 6 or Galaxy S21+ — The Bridges (Optional but Recommended)"

Esto implica que hay teléfonos **dedicados como bridges fijos**, contradiciendo directamente el principio central del protocolo Firefly:
> "We do not have permanent 'Repeaters'. We have roles that rotate." (Doc 01 §3)

### Impacto

Si el equipo de campo lleva "bridges dedicados", intentarán configurarlos manualmente, lo cual no existe en el protocolo. Generará confusión operacional.

### Resolución Propuesta

Eliminar la categoría "Bridges" del Runbook. Si se quieren llevar teléfonos extra, llamarlos "Judge Backup Devices" o "Spare Devices". Su rol de bridge lo decide automáticamente el protocolo.

## INCONSISTENCIA #12 — Generación de Múltiples QR Confusos para el Admin

### El Conflicto

Doc 13 (Admin Dashboard) define **3 QR codes distintos** generados por el admin:
1. "REGISTRATION QR" → para que estudiantes registren proyectos (§1A).
2. "JUDGE QR" (Blue Border) → para que jueces accedan al evento (§1B).
3. "MASTER QR CODE" → resultado de "Generate Event Keys" (§1C).

Doc 04 §2 define el "Project Identification QR" como un 4to tipo.
Doc 06 §3A referencia "JUDGE ACCESS QR".

**¿Son el JUDGE QR (Doc 13) y el JUDGE ACCESS QR (Doc 06) el mismo?** No está claro.
**¿El MASTER QR CODE (Doc 13) es el mismo que el Judge Access QR de Doc 06?** Posiblemente.

### Impacto

Un desarrollador implementando `AdminViewModel` no sabe cuántos QR codes tiene que generar ni cuáles son distintos.

### Resolución Propuesta — Taxonomía Canónica de QR Codes

| Nombre Canónico | Generado por | Escaneado por | Contenido |
|:----------------|:------------|:-------------|:---------|
| **Event Registration QR** | Admin (inicio) | Estudiantes (cámara) | URL local de registro: `http://192.168.x.x:5000/register?event=ID` |
| **Judge Access QR** | Admin (una vez) | Jueces (Nodus App) | `AES_Encrypt({SharedKey, ServerUUID, EventID}, PasswordHash)` |
| **Project Display QR** | Auto (por proyecto) | Jueces (Nodus App al votar) | `nodus://vote?pid=PROJ-A3X&cat=SOFTWARE` |

Solo 3 tipos de QR. No 4 o 5.

## INCONSISTENCIA #14 — Offline-First en Nodus.Web: LocalStorage vs SQLite

### El Conflicto

Doc 03 define `SQLite` como la base de datos offline para todas las apps. Doc 17 e Doc 18 implementan Nodus.Web usando **LocalStorage del navegador** para persistencia.

LocalStorage tiene limitaciones críticas vs SQLite:
- Límite de **5-10MB** (SQLite: sin límite práctico).
- No soporta queries SQL.
- Funciona solo en el hilo principal (no background workers).
- No es transaccional.

### Impacto

Para un evento con 100 proyectos y datos de QR, probablemente sea suficiente. Pero la inconsistencia conceptual entre "todos usan SQLite" y "web usa LocalStorage" no está documentada como una decisión deliberada.

### Resolución Propuesta

Documentar explícitamente que **Nodus.Web usa IndexedDB** (no LocalStorage) vía `Blazored.LocalStorage` o directamente con `IJSRuntime`. IndexedDB tiene capacidad similar a SQLite en browser. O alternativamente, aceptar LocalStorage con un límite documentado de proyectos (~500 proyectos max).

## Arquitectura Final Decidida

```
Nodus.Admin   (Windows MAUI)    ←→ BLE GATT Server + Local HTTP Server
Nodus.Judge   (Android/iOS)     ←→ BLE Swarm Client + Relay
Nodus.Web     (Blazor WASM)     ←→ HTTP/Wi-Fi only (online)
Nodus.API     (ASP.NET Core 10) ←→ HTTPS → MongoDB (cloud sync)
```
